キャッシュ効率: 行優先 vs 列優先

★ HIGHLIGHT

demo_cache_efficiency の中身。 同じ要素を全部足すだけなのに、順番が違うだけで速度が変わる。 CPU がメモリを「64 byte 単位の行 (cache line)」でまとめて持ってくるからだ。

現在の位置
Cache hit
Cache miss
Cache line = 連続する 8 要素 (64 byte)。 行優先なら 1 ライン読めば 8 個ヒット。列優先は 1 個進むごとに次のラインを読み直す。
統計
step
0
cache hit
0
cache miss
0
hit rate

対応コード
# row-major (速い)
for i in range(size):
  for j in range(size):
    s += matrix[i][j]

# col-major (遅い)
for j in range(size):
  for i in range(size):
    s += matrix[i][j]
なぜ速い
Row-major では、ある行の要素は 連続したメモリ に並んでいる。 CPU は最初の要素を取りに行ったついでに後ろの 7 個も ただで 持ってくる。 残り 7 回はキャッシュから一瞬で返る。
なぜ遅い
Col-major は 毎回 size 個飛んだ 場所を読む。 せっかく取ってきた行はほとんど使わずに次の行へ。 結果としてメモリと CPU の間で何度も往復が起き、実測で 5〜10 倍遅くなることもある。

難解なリスト内包表記を分解する

★ HIGHLIGHT

一見ぎゅっと圧縮されたこの 1 行が、実際は 2 重ループと同じこと をしている。 内側 / 外側 / 式が分かれば、どんなネスト内包も読める。

構造を分解
matrix = [
  [i * size + j  # ← (1) 値を作る式
   for j in range(size)]  # ← (2) 内側ループ (列を回す)
  for i in range(size)   # ← (3) 外側ループ (行を回す)
]
(1) 式
i * size + j

行 i, 列 j のセルに入れる値。「行頭からの通し番号」を作っている。

(2) 内側
for j in range(size)

行 i を作るために、列 j を 0..size-1 で回す → 1 行分のリストを返す。

(3) 外側
for i in range(size)

i を 0..size-1 で回し、上で作った「1 行分」を集める → 行のリスト。

A: 内包表記
matrix = [
  [i * size + j for j in range(size)]
  for i in range(size)
]

文字列の罠: O(N²) concat

Python の str は immutable (変更不可)。 result += c は中身を直接変えるのではなく、 毎回 「これまでの全部 + 1 文字」を新しい場所にコピーして作り直す。

Bad += で 1 文字ずつ O(N²) 
result = ""
for c in chars:
    result += c    # 毎回 全コピー
return result
コピー量の積み上がり
縦軸: ループ i 回目に「コピーする文字数」。 1+2+3+…+N ≒ N²/2 → O(N²)。
Good list に append → join O(N) 
parts = []
for c in chars:
    parts.append(c)   # O(1)
return "".join(parts) # O(N)
コピー量の積み上がり
append は配列の末尾に置くだけ。最後の join で 1 回だけ N 文字書き出す → O(N)。

N が増えたとき、計算量がどれだけ 違う速度で爆発するか。 スライダーで N を動かすと、各クラスの「相対操作回数」が動く。

O(log n)対数
binary search
O(n)線形

O(n log n)ソート

Cheat sheet — 1 秒で解ける目安
N の上限 必要な計算量 代表的な手段
N ≤ 10O(N!), O(2ᴺ)全探索 / 順列
N ≤ 20O(2ᴺ)bitmask DP
N ≤ 5,000O(N²)2 重ループ
N ≤ 10⁵O(N log N)ソート + 走査
N ≤ 10⁶O(N)sliding window / hash
N ≤ 10⁹O(log N) / O(1)数学 / 二分探索
N ≥ 10⁵ で O(N²) が見えたら、 sliding window / hashmap / 二分探索 を考える。

感覚で覚える: 各クラスはなぜそうなるか

数式ではなく 絵で殴る。なぜ log n は速く、 n² は爆発し、 n log n は「N 要素 × 木の深さ」になるのか。

分割統治の木: マージソートの深さ

配列を半分・半分に割っていくと、木の 深さは log₂ N。 各レベルでは合計 N 個の要素を比較するので、全体は N × log N。

縦方向 (深さ)
半分に割り続けるので log₂ N 段。
横方向 (各レベル)
どの段も合計 N 個の要素を扱う。
かけ算
縦 × 横 = N × log N。これがソートの下限。
O(n²)

全ペアの爆発: ドットマトリクス

N がちょっと増えただけでドットの数は 二乗で増える。 スライダーを動かすと、N が線形に増えるのに対し、ドット数 (= N²) が爆発する様子が見える。

なぜ N² なのか
for i in range(n):

外ループ n 回 × 内ループ n 回 = n² 回。 重複や自己ペアを除けば n × (n − 1) / 2 ≈ n² / 2 だが、定数倍は無視するので 最終的に O(n²)。
逃げ道: sliding window / hashmap / sort + 二分探索 で O(n log n) や O(n) に落とせるケースが多い。
