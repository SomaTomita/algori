# heapify の仕組み

`heapq.heapify(arr)` は、**バラバラな配列を in-place で heap に並べ替える**操作。
そして驚くべきことに **O(n)** で終わる（要素を 1 個ずつ push する O(n log n) より速い）。

ここでは「どう動くか」と「なぜ O(n) なのか」を、トレース付きで分解する。
配列と二分木の対応（`parent(i)=(i-1)//2`, `left=2i+1`, `right=2i+2`）は
[README.md](./README.md) を前提にする。

---

## 1. 部品: sift-down（1 ノードを正しい位置まで沈める）

heapify の唯一の部品が **sift-down**（「下げ操作」）。
ある位置 `i` の値が子より大きいなら、小さい方の子と入れ替えて 1 段下げる。
それを「子より小さくなる」または「葉に着く」まで繰り返す。

```python
def sift_down(arr, i, n):
    while True:
        smallest = i
        l, r = 2 * i + 1, 2 * i + 2
        if l < n and arr[l] < arr[smallest]:
            smallest = l
        if r < n and arr[r] < arr[smallest]:
            smallest = r
        if smallest == i:      # 子より小さい = もう正しい位置
            break
        arr[i], arr[smallest] = arr[smallest], arr[i]   # 小さい子と入替
        i = smallest           # 入れ替えた先へ降りて続行
```

ポイント: sift-down は **「2 つの子の subtree が既に正しい heap」である前提**で、
root だけがズレている木を直す操作。この前提が次の heapify の順序を決める。

## 2. heapify 本体: 最後の親から root へ「逆順」に sift-down

```python
def heapify(arr):
    n = len(arr)
    for i in range(n // 2 - 1, -1, -1):   # 最後の親 → root へ、逆順
        sift_down(arr, i, n)
```

たった 2 行。効くのは 2 つの設計判断。

### なぜ `n // 2 - 1` から始めるか
index が `n // 2` 以上のノードは **すべて葉**（子を持たない）。
葉は「サイズ 1 の正しい heap」なので sift-down する必要がない。
だから処理が要る最初のノード = 最後の親 = `n // 2 - 1`。

#### まず混乱ポイント: `n`（個数）と index（0 から数えた名前）は別物
- `n` = ノードが **何個あるか**（個数）。`n = len(arr)`。
- index = その中の **何番目か**（名前）。**0 から**数える。

例えばノードが **7 個** なら `n = 7`。でも名前は `0,1,2,3,4,5,6`。
**個数は 7、最後の名前は 6（= `n - 1`）。** ここがズレるのが混乱の元。全部「0 から数える」せい。

| 言葉 | 意味 | 計算 | 値 (n=7) |
|---|---|---|---|
| `n` | ノードの個数 | 数える | 7 |
| `n - 1` | 最後のノードの index | 7 - 1 | 6 |
| `n // 2 - 1` | 最後の親の index | 7÷2=3, 3-1 | 2 |
| 0 | root の index | — | 0 |

#### 木で見る（名前は 0 から。`[9,4,7,1,8,3,6]` n=7 に対応）

```
          (0)          ← root
        /     \
      (1)     (2)      ← 親。(2) が最後の親
     /  \     /  \
   (3) (4)  (5) (6)    ← 葉（子なし）
```

- ノード = 7 個（`n = 7`）。
- 親 = `0, 1, 2`。最後の親 = `2` = `n // 2 - 1`。
- 葉 = `3, 4, 5, 6`（index が `n // 2 = 3` 以上は全部葉）。

heapify は **最後の親（2）から root（0）まで** うしろ向きに直す（`2 → 1 → 0`）。
葉（3〜6）は元から正しい heap なので触らない。だから開始は `n // 2 - 1`。

### なぜ「逆順（下から上）」なのか
sift-down は §1 のとおり「両方の子の subtree が既に heap」を前提にする。
index の大きいノードほど木の下にいる。逆順（大きい index → 小さい index）で処理すれば、
**親を処理する時点で、その子たちの subtree はすでに heap 化済み**が保証される。

もし root から上→下にやると、最初の root を sift-down する時点で子の subtree が
まだ heap になっておらず、1 回の sift-down では直りきらない。だから下から上。

## 3. 完全トレース: `[9, 4, 7, 1, 8, 3, 6]` を min heap 化

最初の配列（まだ heap ではない）:

```
配列: [9, 4, 7, 1, 8, 3, 6]
       0  1  2  3  4  5  6

          9 (i=0)
        /        \
      4 (i=1)     7 (i=2)
     /   \       /   \
   1(3)  8(4)  3(5)  6(6)
```

`n=7` なので最後の親は `7//2 - 1 = 2`。`i = 2, 1, 0` の順に sift-down する。

### i=2 (値 7): 子は 左 3(i=5), 右 6(i=6)
最小の子 3 < 7 なので入替。

```
          9
        /     \
      4       (3)      ← 7 と 3 を交換
     / \      /  \
   1   8    (7)   6
配列: [9, 4, 3, 1, 8, 7, 6]
```
7 は i=5 に降りたが、whileループで、再度 左 `2*5+1=11`・右 `2*5+2=12`を計算され、
> if l < n and arr[l] < arr[smallest]:   # 11 < 7 は False → 入らない
> if r < n and arr[r] < arr[smallest]:   # 12 < 7 も False → 入らない
終了。 (そこは葉なので)
my_heapify関数のfor文に戻り、次のsift_down (i=1) を実行。

### i=1 (値 4): 子は 左 1(i=3), 右 8(i=4)
最小の子 1 < 4 なので入替。

```
          9
        /     \
      (1)      3        ← 4 と 1 を交換
     /  \     /  \
   (4)  8   7    6
配列: [9, 1, 3, 4, 8, 7, 6]
```
4 は i=3 に降りたが、whileループで、再度 左 `2*3+1=7`・右 `2*3+2=8`を計算され、
> if l < n and arr[l] < arr[smallest]:   # 7 < 7 は False → 入らない
> if r < n and arr[r] < arr[smallest]:   # 8 < 7 も False → 入らない
終了。 (そこは葉なので)
my_heapify関数のfor文に戻り、次のsift_down (i=0) を実行。

### i=0 (値 9): 子は 左 1(i=1), 右 3(i=2) ← ここで sift-down が2段カスケード
最小の子 1 < 9 なので入替（9 が i=1 へ降りる）。

```
          (1)
        /      \
      (9)       3        ← 9 と 1 を交換
     /  \      /  \
   4    8    7    6
配列: [1, 9, 3, 4, 8, 7, 6]
```
まだ終わらない。i=1 の 9 を見ると、子は 左 4(i=3), 右 8(i=4)。最小 4 < 9 なので再び入替。

```
          1
        /     \
      (4)      3         ← 9 と 4 を交換
     /  \     /  \
   (9)  8   7    6
配列: [1, 4, 3, 9, 8, 7, 6]
```
9 は i=3 に降りたが、whileループで、再度 左 `2*3+1=7`・右 `2*3+2=8`を計算され、
> if l < n and arr[l] < arr[smallest]:   # 7 < 7 は False → 入らない
> if r < n and arr[r] < arr[smallest]:   # 8 < 7 も False → 入らない
終了。 (そこは葉なので)
my_heapify関数のfor文に戻るが、i=0 が最後（次は -1 で range の外）なので for ループも終了。heapify 完成。

### 完成

```
          1            ← root が最小
        /     \
      4         3
     / \       / \
    9   8     7   6
配列: [1, 4, 3, 9, 8, 7, 6]
```

検算（親 <= 子）: `1<=4,3` / `4<=9,8` / `3<=7,6` すべて OK。正しい min heap。
i=0 の sift-down が **2 段**降りたのに注目（root は深く沈むことがある）。

## 4. なぜ O(n) なのか（核心）

「sift-down を n/2 個のノードでやる、各 sift-down は最大 log n 段 → O(n log n) では?」
と思いがち。でも実際は **O(n)**。理由は「深いノードほど数は多いが、沈む距離は短い」から。

heapify は下から処理するので、各ノードの仕事量は「そのノードの**高さ**（下までの距離）」で決まる。
高さ h のノードは sift-down で最大 h 段しか降りない。そして高さ h のノード数は約 `n / 2^(h+1)`。

| 高さ h | ノード数(約) | 1ノードの最大 swap | この層の合計仕事 |
|-------|------------|------------------|----------------|
| 0 (葉) | n/2  | 0 | 0 |
| 1     | n/4  | 1 | n/4 |
| 2     | n/8  | 2 | 2n/8 |
| 3     | n/16 | 3 | 3n/16 |
| ...   | ...  | ...| ... |

合計仕事 ≈ `n × Σ(h / 2^(h+1))`。この級数 `Σ h/2^h`（h=0,1,2,...）は **2 に収束**する。
よって合計 ≈ `n × 定数 = O(n)`。

直感で言うと:
- **葉（全体の半分）は仕事ゼロ。** 一番数が多いノードが一番ラク。
- 仕事が重いのは root 付近の**ごく少数**のノードだけ。
- 「多いノードに軽い仕事、少ないノードに重い仕事」を割り当てているので合計が抑えられる。

### 対比: なぜ「push を n 回」だと O(n log n) なのか
空の heap に 1 個ずつ push する作り方は、各要素を**葉から root へ sift-up**する。
このとき重い仕事（最大 log n 段の上昇）が、**数の多い葉側のノード**にかかる。
`n/2` 個のノード × 最大 `log n` 段 = O(n log n)。

つまり同じ「heap を作る」でも、**仕事の重い方向を「多いノード」に向けるか「少ないノード」に
向けるか**で計算量が変わる。heapify（下から sift-down）は後者なので速い。

```
heapify (下から sift-down)  : 重い仕事 → root 付近の少数  → O(n)
push を n 回 (sift-up)       : 重い仕事 → 葉側の多数      → O(n log n)
```

## 5. 自分で確かめる（コピペで動く）

```python
def sift_down(arr, i, n):
    while True:
        smallest, l, r = i, 2 * i + 1, 2 * i + 2
        if l < n and arr[l] < arr[smallest]:
            smallest = l
        if r < n and arr[r] < arr[smallest]:
            smallest = r
        if smallest == i:
            break
        arr[i], arr[smallest] = arr[smallest], arr[i]
        i = smallest

def my_heapify(arr):
    n = len(arr)
    for i in range(n // 2 - 1, -1, -1):
        sift_down(arr, i, n)
        print(f"i={i} 処理後: {arr}")

import heapq
a = [9, 4, 7, 1, 8, 3, 6]
b = a.copy()
my_heapify(a)
heapq.heapify(b)
print("自作    :", a)
print("heapq   :", b)   # 同じ並びになる（同じアルゴリズム）
```

`heapq.heapify` は C 実装だが、やっていることはこの下から sift-down と同一。

## 6. Python `heapq` と、符号反転 max heap への接続

`heapq` は min heap 専用なので、**max heap が欲しいときは全要素の符号を反転**して
heapify する（[README.md](./README.md) の定番ハック）。

この `[ -n for n in nums ]` → `heapq.heapify` の動きを実例で追ったのが
[kth_largest_element.py](./kth_largest_element.py) の解法 1 のトレース。
そこで出てくる `[-3,-2,-1,-5,-6,-4] → [-6,-5,-4,-3,-2,-1]` という並べ替えは、
本ファイル §3 と全く同じ「下から sift-down」を負の値に対してやっているだけ。
（min heap の root = 最小の負数 = 元の最大値、という対応）

## まとめ

- heapify = **最後の親から root へ逆順に sift-down** するだけ。
- 逆順なのは、親を直す時点で子の subtree が heap 済みである必要があるから。
- 計算量は **O(n)**。深いノードは多いが沈む距離が短く、合計が `n × 定数` に収まる。
- 1 個ずつ push する O(n log n) より速いので、最初に全要素が揃っているなら heapify 一択。
